# vscode-msr Alias User Guide

This guide helps you use the command-line aliases generated by the **vscode-msr** extension. These aliases wrap [msr](https://github.com/qualiu/msr) (regex search tool) and [nin](https://github.com/qualiu/msr) (set operations tool) to provide fast, powerful code searching and common development operations directly from your terminal.

> **Looking for msr/nin tool documentation?** See [msr User Guide](msr-user-guide.md), [nin User Guide](nin-user-guide.md), and [Download Links](download-links.md).

> **Practical use cases and AI integration?** See [Use Cases and Comparisons](use-cases-and-comparisons.md) and [AI Agent Usage Guide](ai-agent-usage-guide.md).
>

## Quick Start

1. Install the **vscode-msr** extension in VS Code.
2. Open a terminal in VS Code — aliases are automatically loaded.
3. Start searching:

```bash
# Search for "MyClass" in all C++ files under current directory
find-cpp -t MyClass

# Search for "MyClass" definition in C++ files (first arg = search term)
find-cpp-def MyClass

# Search using git-tracked files (respects .gitignore)
gfind-cpp -t MyClass

# Don't know the alias name? Discover it:
find-alias find-cpp
```

> **Tip**: You don't need to memorize all aliases. Use `find-alias` to discover what's available — see [Self-Discovery with find-alias](#self-discovery-with-find-alias).

## Self-Discovery with find-alias

The `find-alias` command is your primary tool for discovering aliases:

```bash
# Find aliases by name prefix
find-alias find-cpp

# Find all git-related aliases
find-alias git

# Find aliases matching a regex pattern
find-alias "^gfind"

# Show names only (compact view)
find-alias docker -Output Name

# Find aliases by description keyword
find-alias -Description "submodule"
```

`find-alias` searches both your VS Code settings and alias files (doskeys/bashrc), showing alias name, body, and description.

For any alias you're unsure about, run `find-alias <alias-name>` to check if it exists and what it does. For example: `find-alias gfind-py-def`.

## Naming Conventions

### Scope Prefixes: find → gfind → rgfind

Aliases follow a consistent naming pattern with three scope levels:

| Prefix         | Scope                         | Description                                                              |
| -------------- | ----------------------------- | ------------------------------------------------------------------------ |
| `find-{ext}`   | Current directory (recursive) | Searches all matching files under `.`, excluding junk paths              |
| `gfind-{ext}`  | Git repository                | Searches only git-tracked files (`git ls-files`), auto-detects repo root |
| `rgfind-{ext}` | Multiple git repos            | Searches across sibling directories that are independent git repos       |

**Examples**:
- `find-py -t "pattern"` → search content in all `*.py` files recursively from current directory
- `gfind-py -t "pattern"` → search content in git-tracked `*.py` files in the current repo
- `rgfind-py -t "pattern"` → search content in `*.py` across sibling directories that each have `.git`

The `gfind-*` aliases use **git hash-based caching** — the file list is automatically refreshed when the git commit changes (branch switch, pull, new commit, etc.).

> **Tip**: In a git repository, **prefer `gfind-*` over `find-*`**. `gfind-*` only searches git-tracked files, automatically excluding build output, `node_modules`, and other `.gitignore` entries — giving cleaner, more relevant results. Use `find-*` when you need to search newly created files not yet tracked by git.

> **Note**: `rgfind-*` aliases are disabled by default (except `rgfind-file`). See [Controlling Alias Generation](#controlling-alias-generation) to enable them.

### Suffixes: -def and -ref

| Suffix           | Purpose                                                                               |
| ---------------- | ------------------------------------------------------------------------------------- |
| `find-{ext}-def` | Search **definitions** (classes, functions, methods) using language-specific patterns |
| `find-{ext}-ref` | Search **references** (usages) using language-specific patterns                       |

Definition aliases (`-def`) are only generated for languages with specific definition patterns — not for generic formats like XML or JSON.

**Examples**:
- `find-cpp-def MyClass` → find where `MyClass` is defined in C++ files
- `find-java-ref myMethod` → find references to `myMethod` in Java files
- `gfind-py-def my_function` → find definition in git-tracked Python files

### Passing Extra Arguments

There are two argument conventions depending on the alias type:

**Base aliases** (`find-{ext}`, `find-code`, `find-all`, `find-file`, etc.) pass **all arguments directly to msr** via `$*`. You must provide a match flag (`-t`, `-x`, `--nt`, or `--nx`) for content search:

> ⚠️ **Warning**: If you omit the match flag (e.g., run `find-py` with no arguments), msr outputs **all lines of all matching files**, flooding your terminal with massive output.

```bash
# Content search: must provide -t or -x flag explicitly
find-cpp -t MyClass
find-py -t "import os"
find-code -t "TODO"
find-cs -x "TODO"              # -x is faster than -t for plain text (no regex overhead)

# Case-insensitive match
find-cpp -t MyClass -i

# Show only first 10 results
find-py -t my_function -H 10

# Search and replace (preview)
find-java -t oldName -o newName

# Pipe to other commands
find-ts -t myFunc | msr -t "import"
```

**`-def` and `-ref` aliases** (`find-{ext}-def`, `find-{ext}-ref`, `find-def`, `find-ref`, `find-class`, etc.) use the **first argument as the search term** (embedded in the built-in `-t` pattern via `$1`). Extra arguments are appended:

```bash
# First arg = search term (automatically used in -t pattern)
find-cpp-def MyClass
find-py-ref my_function
find-def "MyClass"
find-class "MyClass"

# Extra args after the search term
find-cpp-def MyClass -i
find-java-ref MyInterface -H 20
```

## Common Workflows

### Searching Code by Language

```bash
# Search in specific language files (must provide -t for content search)
find-cpp -t "pattern"       # C/C++ files (.cpp, .cxx, .cc, .c, .h, .hpp)
find-java -t "pattern"      # Java files (.java, .scala, .groovy)
find-py -t "pattern"        # Python files
find-cs -t "pattern"        # C# files (.cs, .cshtml)
find-go -t "pattern"        # Go files
find-ts -t "pattern"        # TypeScript files
find-js -t "pattern"        # JavaScript files

# Search across broader categories
find-code -t "pattern"      # All code files
find-doc -t "pattern"       # Documentation files
find-config -t "pattern"    # Configuration files
find-script -t "pattern"    # Script files (.sh, .bat, .cmd, .ps1)
find-all -t "pattern"       # All files (no size limit)
```

### Universal Generic Aliases: find-file and find-small

These aliases have **no built-in file type filter (`-f`) or content match (`-t`)**. All arguments pass directly to msr. They come in three scope variants each:

| Alias group                                   | Scope                  | Description                                          |
| --------------------------------------------- | ---------------------- | ---------------------------------------------------- |
| `find-file` / `gfind-file` / `rgfind-file`    | dir / git / multi-repo | Universal generic — no `-f`, no `-t`, no size limits |
| `find-small` / `gfind-small` / `rgfind-small` | dir / git / multi-repo | All file types, size limited to 1B–1.6MB             |

**Why use them?**

- **`find-file`**: Search any file type ad-hoc without needing a `find-{ext}` alias. Just add `-f` to specify the file type on the fly. `rgfind-file` is **always enabled** — no configuration needed.
- **`find-small`**: A safer alternative to `find-all` for broad searches — the 1.6MB size cap skips large generated/binary files.

```bash
# find-file: search any file type without a dedicated alias
find-file -f "\.proto$" -t "pattern"  # Search .proto files (no find-proto alias needed)
find-file -f "\.config$" -l           # List files by name pattern
find-file -t "TODO"                   # Search content across all files
gfind-file -f "\.json$" -l           # List json files in git repo
rgfind-file -f "\.yaml$" -t "key:"   # Search yaml across sibling repos

# find-small: broad search with size safety
find-small -t "TODO"                  # Search all small files (1B–1.6MB)
gfind-small -t "deprecated" -i       # Case-insensitive, git-tracked only
```

> ⚠️ **Warning**: Like all base aliases, omitting `-t` (or other match flags) causes msr to output **all lines of all files** — use `-t`, `-x`, `--nt`, or `--nx` to filter content.

### Searching Definitions and References

```bash
# Find where a symbol is defined
find-cpp-def MyClass
find-py-def my_function
find-java-def MyInterface

# Find where a symbol is referenced
find-cpp-ref MyClass
find-py-ref my_function
find-java-ref MyInterface

# Cross-language searches
find-def "MyClass"          # Definitions in all file types
find-ref "MyClass"          # References in all file types
find-all-def "MyClass"      # All definitions
find-pure-ref "MyClass"     # References excluding definitions

# Specialized searches
find-class "MyClass"        # Class/struct/enum/interface/trait definitions
find-member-ref "myField"   # Member references
find-spring-ref "myBean"    # Java Spring references
```

> **Tip**: `-def` aliases use complex regex that can be slow or return no results. **Recommended alternatives**:
>
> ```bash
> # Use -ref with -x to filter for definition keywords (more reliable)
> gfind-py-ref MyClass -x class      # Find references containing "class" literally
> gfind-ts-ref myFunc -x function    # Find references containing "function" literally
>
> # Or use base alias with custom regex
> gfind-py -t "class\s+MyClass\b"    # Direct regex for class definition
>
> # Add -d / --sp / --xp to narrow scope further
> gfind-cs-ref MyService -x class -d Services
> ```

### Git-Aware Searching

```bash
# Search only git-tracked files (respects .gitignore)
gfind-cpp -t "pattern"
gfind-py-def my_function
gfind-java-ref MyClass

# gfind-file: all args go directly to msr (same as find-file but git-scoped)
gfind-file -f "\.json$" -l

# gfind-* automatically:
# - Detects git repo root
# - Uses cached file list (refreshed on git hash change)
# - Excludes files in .gitignore
```

### Multi-Repo Searching

For workspaces with multiple sibling git repositories:

```
workspace/
├── repo1/    (.git)
├── repo2/    (.git)
└── repo3/    (.git)
```

```bash
# rgfind-file: all args go directly to msr (always available)
rgfind-file -f "\.yaml$" -l

# If enabled via settings, search by language across repos
rgfind-cpp -t "pattern"
rgfind-java-def MyClass
```

### File Distribution and Sorting

```bash
# File distribution statistics
find-top-folder             # Top folders by file count (all files)
find-top-type               # Top file types (all files)
find-top-source-folder      # Top folders (source files only)
find-top-source-type        # Top file types (source files only)
find-top-code-folder        # Top folders (code files only)
find-top-code-type          # Top file types (code files only)

# Git-tracked file distribution (gfind-top-* variants)
gfind-top-folder            # Top folders (git-tracked files only)
gfind-top-type              # Top file types (git-tracked files only)
gfind-top-code-type         # Top code file types (git-tracked)

# Sort files
sort-by-size                # Sort all files by size
sort-by-time                # Sort all files by modification time
sort-source-by-size         # Sort source files by size
sort-source-by-time         # Sort source files by time
sort-code-by-size           # Sort code files by size
sort-code-by-time           # Sort code files by time

# Quick sort shortcuts
sfs                         # Sort files by size
sft                         # Sort files by time
```

### Git Diff Against Main Branch

```bash
gdm                         # Difftool current branch against main/master
gdm-l                       # List all changed files vs main/master
gdm-al                      # List added files vs main/master
gdm-ml                      # List modified files vs main/master
gdm-dl                      # List deleted files vs main/master
gdm-m                       # Difftool modified files only vs main/master
gdm-nt                      # Diff excluding test files

# Combining git diff with msr pipes for deeper analysis:
git diff origin/main | msr -t "^\+" --nt "^(\+\+\+|---)" -x "TODO"   # Find newly added TODOs
git diff origin/main | msr -t "^\+" -x "console.log"                  # Check for leftover debug code
```

### Git History Search

```bash
git-find-commit "fix bug"       # Search commit messages
git-find-content "MyClass"      # Search changed content across commits
git-find-log "release"          # Search git log
git-find-creation "file.txt"    # Search file creation history
git-find-deletion "old-module"  # Search file deletion history
git-find-update "config"        # Search file update history
```

## Go to Definition and Find References

vscode-msr provides intelligent **Go to Definition** (`F12`) and **Find References** (`Shift+F12`) that work across any language — no language server required.

### How It Works

When you press `F12` on a symbol, vscode-msr:

1. **Extracts the word** under the cursor
2. **Searches progressively** from nearest to farthest scope (same file → sibling files → same module → entire repo)
3. **Scores results** using a multi-dimensional system (path relevance, public/private modifiers, test vs production code)
4. **Returns the best match** — typically in under 200ms

### Progressive Search Strategy

vscode-msr searches in expanding scope groups, stopping as soon as results are found:

| Group | Scope | Typical Speed |
|-------|-------|---------------|
| 1 | Same file | 5–70ms |
| 2 | Sibling files (`-k 1`) | 50–150ms |
| 3 | Same module (`--sp` path filter) | 80–200ms |
| 4 | Entire repository | 200–500ms |

> **Performance note**: Most lookups hit Groups 1–3, staying under 150ms. Optimization parameters (`--sp`, `--xp`, `-H 1 -J`) are applied automatically.

### Context-Aware Scoring

Results are ranked by a multi-dimensional scoring system:

| Factor | Score Boost | Example |
|--------|-------------|---------|
| Non-test path | +500 | `src/` ranks higher than `test/` |
| Public modifier | +200 | `public class` ranks higher than `internal class` |
| Namespace match | +300 | Matches `using` declarations in the current file |
| Path proximity | +100~400 | Files closer to the current file score higher |

**Example — Multiple Same-Name Classes:**

```csharp
// Current file has: using MyApp.Services;
// Pressing F12 on "Logger" finds MyApp.Services.Logger first
// NOT MyApp.Tests.MockLogger, NOT ThirdParty.Logger
```

vscode-msr reads `using`/`import` declarations and filters results by referenced namespaces — a feature traditional language servers don't always provide.

### When to Use vscode-msr's F12

| Scenario | Traditional LSP | vscode-msr |
|----------|----------------|------------|
| First lookup after repo clone | 30s+ (waiting for index) | 200–400ms (no index needed) |
| Memory footprint | 1–4GB for index | ~50MB (no persistent index) |
| Cross-language projects | Requires LSP per language | Works immediately via regex |
| Production code vs test mocks | No preference | Production code ranked higher |

**Best for:**
- Large monorepos (100K+ files) where LSP indexing is slow
- Cross-language projects without full LSP setup
- Quick navigation in unfamiliar codebases
- Finding definitions when test mocks clutter LSP results

### Usage Tips

```bash
# If F12 doesn't find the right definition, fall back to command-line aliases:
find-cpp-def MyClass              # Search definitions in C++ files
gfind-py-ref my_function          # Search references in git-tracked Python files

# Narrow results with extra flags:
gfind-cs-ref Logger -x "using"    # References containing "using" literally
```

> **Tip**: vscode-msr's F12 and the `find-xxx-def` command-line aliases use the same underlying regex patterns and scoring logic.

## How Aliases Are Activated

- **VS Code terminals**: Aliases are loaded automatically when you open a terminal.
- **External terminals**: Run `update-alias` or `use-this-alias` in an external terminal to load aliases.
- **Alias files**: On Windows, aliases are stored in `%USERPROFILE%\msr-cmd-alias.doskeys`. On Linux/macOS, they are in `~/msr-cmd-alias.bashrc`. See [Common-Alias.md — Alias Loading](https://github.com/qualiu/vscode-msr/blob/master/Common-Alias.md#alias-loading-for-different-terminal-types) for the full terminal-to-file mapping.

> **Important for git repos**: **Open the repo directory in VS Code** to automatically generate aliases and repo-specific environment variables (`$Skip_Junk_Paths`, `$Skip_Junk_Name`). When properly initialized, `find-*` aliases use `--np "git ignore path regex pattern"` (derived from `.gitignore`) for precise exclusion. Without initialization, they fall back to the default `--nd "junk folder regex pattern"` from the `msr.default.skipFolders` setting (configurable). For external terminals, run `use-this-alias` to load already-generated repo-specific aliases (requires VS Code to have opened the repo first).

Supported terminal types: CMD, PowerShell, Pwsh, Bash (Linux/macOS), WSL, MinGW (Git Bash), Cygwin.

## Managing Aliases

### Finding Aliases

```bash
# By name prefix
find-alias find-

# By regex pattern (auto-detected when special chars present)
find-alias "^gfind"
find-alias "find-.*-def$"

# Names only (compact)
find-alias docker -Output Name

# By description
find-alias -Description "environment"
```

### Removing Aliases

```bash
# Remove a single alias
rm-alias my-custom-alias

# Remove multiple aliases (comma-separated)
rm-alias rgfind-xml,rgfind-json

# Remove by regex pattern
rm-alias "^rgfind"
rm-alias "find-.*-def$"

# Preview mode (see what would be removed without deleting)
rm-alias "^rgfind" -Preview 1

# Remove from settings.json only
rm-alias my-alias -SettingsOnly 1

# Keep script files during removal
rm-alias my-alias -KeepScripts 1
```

## Configuration

### Custom User Aliases

Add your own aliases in VS Code `settings.json`:

```json
{
  "msr.commonAliasNameBodyList": [
    {
      "aliasName": "my-search",
      "aliasBody": "msr -rp . -f \"\\.ts$\" -t \"$1\" $*",
      "description": "Search in TypeScript files"
    }
  ],
  "msr.cmd.commonAliasNameBodyList": [],
  "msr.bash.commonAliasNameBodyList": []
}
```

- `msr.commonAliasNameBodyList` — aliases for all terminal types
- `msr.cmd.commonAliasNameBodyList` — Windows-only aliases
- `msr.bash.commonAliasNameBodyList` — Linux/macOS-only aliases

### Controlling Alias Generation

| Setting                                             | Purpose                                          | Default                                      |
| --------------------------------------------------- | ------------------------------------------------ | -------------------------------------------- |
| `msr.cookFindDefinitionAlias.extraExtensionPattern` | Extra file extensions to generate `-def` aliases | `""` (auto-detect only)                      |
| `msr.cookAlias.recursiveGitFindExtensionPattern`    | File extensions to generate `rgfind-*` aliases   | `""` (disabled; `rgfind-file` always exists) |
| `msr.cookCmdAlias.outputFullPath`                   | Use full paths in alias output                   | `false`                                      |

**Quick examples**:
- Enable `rgfind-*` for C++ and Java: set `msr.cookAlias.recursiveGitFindExtensionPattern` to `"cpp|java"`
- Enable `rgfind-*` for ALL types: set it to `"."`
- Enable `find-xxx-def` for ALL types: set `msr.cookFindDefinitionAlias.extraExtensionPattern` to `"."`

### New Terminal Alias Initialization

These settings control whether vscode-msr initializes project alias/doskeys when a **new terminal** is opened.

| Setting | Purpose | Default |
| --- | --- | --- |
| `msr.initProjectCmdAliasForNewTerminals` | Initialize project command shortcuts (alias/doskeys) for newly opened terminals. | `true` |
| `msr.skipInitCmdAliasForNewTerminalTitleRegex` | Skip initialization when terminal title matches this regex. | `Integration\|Console\|Extension\|Java\|Python\|Debug` |

Behavior summary:
- Initialization is checked on terminal open.
- If terminal title matches `msr.skipInitCmdAliasForNewTerminalTitleRegex`, initialization is skipped.
- If `msr.initProjectCmdAliasForNewTerminals` is `true`, vscode-msr initializes alias for supported shell-like terminals (CMD/PowerShell/bash/wsl) in normal project scenarios.

Example override in `settings.json`:
```json
{
  "msr.initProjectCmdAliasForNewTerminals": true,
  "msr.skipInitCmdAliasForNewTerminalTitleRegex": "Integration|Debug|Java|Python"
}
```
